## 🚩 阶段一：挑战

> **只看这个阶段，自己想 10 分钟。不要往下翻。**

### 场景：设计一台"读写机器人"

1936 年，24 岁的艾伦·图灵思考了一个问题：

> **"什么是'可计算'？"**

这不是在问"这台计算机能算什么"——当时还没有现代计算机。他问的是：**有没有一个方法能精确地定义"什么东西在原则上可以被计算出来"？**

他抽象出了一个极其简单的模型。现在轮到你来设计它。

### 你的任务

你有一台机器，它只有三样东西：

```
① 一条无限长的纸带，被分成一格一格
   ...| 0 | 1 | 1 | 0 | □ | □ | □ |...
   每个格子可以写一个符号（比如 0、1、或空白 □）

② 一个读写头
   对准纸带上的一个格子，可以：
   - 读取当前格子的符号
   - 写入一个符号覆盖它
   - 向左移一格，或向右移一格

③ 一个有限的状态寄存器
   机器可以处于有限个"状态"之一（比如 "开始"、"找到一"、"回退"……）
   状态就是机器的"瞬时记忆"。
```

机器的行为由一条条**规则**控制，每条规则长这样：

```
如果 当前状态 = S  且  读到符号 = X
那么 → 写入符号 Y、向右/左移、进入状态 S'
```

**你要做的事：**

设计一组规则，让这台机器能**判断一个二进制字符串是不是回文（正读反读一样）**。

比如 `11011` → 是回文，`10100` → 不是。

```
纸带初始状态：
...| 1 | 1 | 0 | 1 | 1 | □ | □ |...
    ↑
    读写头从这里开始，机器处于状态 "start"
```

机器最终应该在纸带上写一个 `Y`（是回文）或 `N`（不是回文），然后停机。

> 💬 *先别去搜索"图灵机"的定义。从零开始想——如果你是这台读写机器人，你会用什么策略来判断回文？可以把这组规则写成一张表格。*
>
> *关键是：你只有有限个状态（不能记住整个字符串），只能一格一格地读和写。*

## 🪜 阶段二：提示链

> **给自己最后一次机会——先试再想 5 分钟。还是没思路？逐条看提示，看到哪条有感觉了就停下来继续想。**

**提示 1：人是怎么判断回文的？**

给定一个字符串 `11011`，你一眼就能看出来是不是回文——因为你的大脑可以同时"看到"整个字符串。

但读写机器人一次只能看一个格子。**你必须把"同时看到整个"这个能力拆成"一步步的操作"**。

如果你蒙上眼睛，只用手摸一串凸起的盲文：

```
左手放在最左端，右手放在最右端。
如果两端的字符相同 → 左手右移一格，右手左移一格，继续比较。
如果不相同 → 不是回文。
如果左右手相遇或交叉 → 是回文。
```

读写机器人可以做同样的事——但它只有"一只手"（一个读写头）。它怎么同时记住两端的信息？

**提示 2：用纸带本身做"两只手"**

机器人没有两只手，但它可以在纸带上**做标记**。

策略：
```
1. 把最左端的字符"记住"（进入状态）
2. 把那个格子标记为 X（已读）
3. 向右走到纸带末尾
4. 把最右端的字符和"记住"的字符比较
5. 如果相同 → 标记这个格子为 X，然后向左走回起点
6. 如果不同 → 写 N，停机
7. 重复，直到所有字符都被标记 → 写 Y，停机
```

示例流程，输入 `101`：

```
初始：[1][0][1][□]  状态: start
                                   ← 这需要你设计具体的状态

步骤1: 读左端 = 1，记住它（进入状态 remembered_1）
       标记为 X
       X│0│1│□   状态: remembered_1

步骤2: 向右走到末尾
       X│0│1│□ → X│0│1│□ → X│0│1│□
         ↑         ↑         ↑
       走到 □ 前一个非空格就是右端

步骤3: 读右端 = 1，和 remembered_1 匹配 ✓
       标记为 X
       X│0│X│□   状态: matched

步骤4: 向左走回左端
       ...

步骤5: 读左端 = 0，记住它（状态 remembered_0）
       标记为 X
       X│X│X│□

步骤6: 向右走到右端，读 = X（已标记）
       说明所有字符都处理完了 → 写 Y，停机
```

**提示 3：把你需要的"规则"写成一张表**

你需要的每个 "状态 + 读到什么 → 做什么" 就是一条规则。

回文检查需要一个比较复杂的规则表——这正是图灵机的力量所在。但我们可以从最简单的开始：

**先实现一个"二进制加 1"的机器**（这个简单很多）：

```
输入：1011（二进制数，最高位在左边）
输出：1100（加 1 后的结果）

策略：从最右端开始，把末尾的 1 变成 0 并向左进位，
      直到遇到 0 → 变成 1，然后停机。
```

| 状态 | 读 | 写 | 移 | 下一状态 |
|:----:|:--:|:--:|:--:|:--------:|
| start | 0/1 | 同 | → | right |
| start | □  | ←  | ←  | carry  |
| right | 0/1 | 同 | → | right |
| right | □  | ←  | ←  | carry  |
| carry | 0  | 1  | ←  | halt   |
| carry | 1  | 0  | ←  | carry  |
| carry | □  | 1  | —  | halt   |

咦，把它换成"如果……那么……"的格式：

```
如果状态 = start 且 读 = 0 → 写 0，右移，状态 right
如果状态 = start 且 读 = 1 → 写 1，右移，状态 right
如果状态 = start 且 读 = □ → 写 □，左移，状态 carry
...
```

看出来了？你刚刚写的就是**图灵机的指令表**。

**提示 4：你的机器 = 一个图灵机**

恭喜你——你刚刚从零设计出了图灵机。

任何符合这三条的东西都叫图灵机：
1. **无限长纸带**：存储器，无限但每次只读写一格
2. **有限状态**：你的"状态"数量有限（不能无穷多个）
3. **确定的转移规则**：给定 (状态, 读到的符号)，确定下一步 (写什么, 往哪移, 进入什么状态)

## 🧠 阶段三：核心洞察 + 三步翻译法

### 核心洞察

> **图灵机是人类能想象到的最简单的通用计算模型——但它能计算一切可计算的东西。**
>
> 一个"可计算"的问题，就是**存在一台图灵机能够解决它**的问题。没有什么计算机比图灵机更强大（Church-Turing 论题）。

这意味着三件让人头皮发麻的事：

**① 你的笔记本电脑并不比图灵机更强大**
任何你的电脑能算的东西，图灵机也能算（只是慢得多）。反过来也成立——图灵机能算的，你的电脑也能算。

**② 但有些事情，图灵机算不了——任何计算机都算不了**
最著名的例子：**停机问题**——给定一个程序，判定它最终会停止还是永远运行下去。
不存在任何图灵机（也不存在任何计算机程序）能解决这个问题。这是可计算性的**边界**。

**③ 生命游戏 = 图灵机 = 你的电脑**
102 中说的"生命游戏是图灵完全的"就在说这个——你能在生命游戏的网格里搭建出一台图灵机，从而计算一切可计算的问题。

```
一台图灵机 = 一条纸带 + 一个读写头 + 一张规则表
              ↓             ↓            ↓
             内存          CPU       指令集(程序)

最简单的模型 = 和现代计算机等价的计算能力
```

### 三步翻译——自己写代码

**第一步：说清意图**

```
写一个图灵机模拟器：
输入：规则表（状态→转移）、初始纸带、初始状态
输出：最终纸带内容和停机状态

再用这个模拟器实现一个"二进制加 1"的图灵机。
```

**第二步：拆成标准操作**

```
→ 定义规则表结构：{(状态, 读): (写, 移, 下一状态)}
→ 初始化纸带（列表）、读写头位置、当前状态
→ 主循环：
    → 读取当前格子符号
    → 查表：(当前状态, 读) → (写, 移, 下一状态)
    → 写符号
    → 移动读写头
    → 更新状态
    → 如果状态 = "halt" → 停机
→ 打印最终纸带
```

**第三步：匹配到代码**

| 标准操作 | 代码实现 |
|---------|---------|
| 规则表 | `transitions = {(state, read): (write, move, next_state)}` |
| 读当前格 | `symbol = tape[head]` |
| 查转移 | `write, move, next = transitions[(state, symbol)]` |
| 写 | `tape[head] = write` |
| 移 | `head += 1 if move == 'R' else -1` |
| 纸带无限 | `if head < 0: tape.insert(0, '□'); head = 0` |
| 停机 | `while state != 'halt':` |

## ✅ 阶段四：参考实现 + 迁移 + 本周挑战

### 参考实现

```python
class TuringMachine:
    """
    通用图灵机模拟器。
    
    规则表格式：
    {(状态, 读符号): (写符号, 移动方向, 下一状态)}
    移动方向：'L'=左, 'R'=右, 'N'=不动
    """
    def __init__(self, tape: list[str], transitions: dict,
                 start_state: str = 'start',
                 blank: str = '□'):
        self.tape = list(tape)       # 纸带
        self.head = 0                # 读写头位置
        self.state = start_state     # 当前状态
        self.transitions = transitions
        self.blank = blank
        self.halted = False

    def step(self):
        """执行一步。"""
        symbol = self.tape[self.head]
        key = (self.state, symbol)

        if key not in self.transitions:
            self.halted = True
            return

        write, move, next_state = self.transitions[key]

        # 写入
        self.tape[self.head] = write

        # 移动读写头
        if move == 'R':
            self.head += 1
            if self.head >= len(self.tape):
                self.tape.append(self.blank)
        elif move == 'L':
            self.head -= 1
            if self.head < 0:
                self.tape.insert(0, self.blank)
                self.head = 0

        # 更新状态
        self.state = next_state

    def run(self, max_steps: int = 1000):
        """运行直到停机或超过最大步数。"""
        steps = 0
        while not self.halted and steps < max_steps:
            self.step()
            steps += 1
        return self.tape, self.state, steps


# ── 示例 1：二进制加 1 ──
def binary_increment():
    """图灵机：二进制数加 1（最低位在右边）。"""
    transitions = {
        # 阶段 1：走到最右端
        ('start', '0'):  ('0', 'R', 'start'),
        ('start', '1'):  ('1', 'R', 'start'),
        ('start', '□'): ('□', 'L', 'carry'),
        # 阶段 2：进位
        ('carry', '0'):  ('1', 'L', 'halt'),
        ('carry', '1'):  ('0', 'L', 'carry'),
        ('carry', '□'): ('1', 'N', 'halt'),
    }

    tape = list("1011")  # 二进制 11
    tm = TuringMachine(tape, transitions)
    final_tape, state, steps = tm.run()

    print(f"二进制加 1：")
    print(f"  输入: 1011 (=11)")
    print(f"  输出: {''.join(final_tape)} (=12)")
    print(f"  步数: {steps}")
    # → 1100


# ── 示例 2：回文检查 ──
def palindrome_checker():
    """图灵机：检查二进制字符串是否为回文。"""
    transitions = {
        # === 阶段 1：找到左端，记住字符 ===
        ('start', '0'):  ('X', 'R', 'find_right_0'),   # 记住 0，标记
        ('start', '1'):  ('X', 'R', 'find_right_1'),   # 记住 1，标记
        ('start', 'X'):  ('X', 'R', 'start'),           # 跳过已标记
        ('start', '□'): ('Y', 'N', 'halt'),            # 全是 X → 空字符串/已完成 → 回文

        # === 阶段 2：向右走到最右端 ===
        ('find_right_0', '0'):  ('0', 'R', 'find_right_0'),
        ('find_right_0', '1'):  ('1', 'R', 'find_right_0'),
        ('find_right_0', 'X'):  ('X', 'R', 'find_right_0'),
        ('find_right_0', '□'): ('□', 'L', 'check_0'),  # 到右端，准备检查

        ('find_right_1', '0'):  ('0', 'R', 'find_right_1'),
        ('find_right_1', '1'):  ('1', 'R', 'find_right_1'),
        ('find_right_1', 'X'):  ('X', 'R', 'find_right_1'),
        ('find_right_1', '□'): ('□', 'L', 'check_1'),

        # === 阶段 3：检查右端 ===
        ('check_0', '0'):  ('X', 'L', 'go_left'),   # 匹配！
        ('check_0', '1'):  ('N', 'N', 'halt'),      # 不匹配 → 不是回文
        ('check_0', 'X'):  ('X', 'L', 'check_0'),   # 跳过已标记
        ('check_0', '□'): ('Y', 'N', 'halt'),      # 只有 X → 已完成

        ('check_1', '1'):  ('X', 'L', 'go_left'),   # 匹配！
        ('check_1', '0'):  ('N', 'N', 'halt'),      # 不匹配
        ('check_1', 'X'):  ('X', 'L', 'check_1'),
        ('check_1', '□'): ('Y', 'N', 'halt'),

        # === 阶段 4：向左走回左端 ===
        ('go_left', '0'):  ('0', 'L', 'go_left'),
        ('go_left', '1'):  ('1', 'L', 'go_left'),
        ('go_left', 'X'):  ('X', 'L', 'go_left'),
        ('go_left', '□'): ('□', 'R', 'start'),     # 回到左端
    }

    def test(s: str):
        tm = TuringMachine(list(s), transitions)
        final_tape, state, steps = tm.run()
        result = ''.join(final_tape).replace('□', '').replace('X', '')
        is_pal = 'Y' in result or 'N' not in result
        print(f"  '{s}' → {'✅ 回文' if is_pal else '❌ 不是'} (步数: {steps})")

    print("回文检查：")
    test("101")    # 回文
    test("11011")  # 回文
    test("100")    # 不是
    test("1")      # 回文

# 下面是一个简化版：一台图灵机读取另一台图灵机的规则并模拟它。
def universal_tm_demo():

    演示"一台图灵机模拟另一台"的概念。
    
    我们用 Python 函数作为"通用图灵机"，
    它接受一个规则表作为输入，执行它。
    关键在于：规则表本身只是符号串——可以被编码在纸带上。
    """
    print("\n通用图灵机概念：")
    print("  一台图灵机可以模拟任意其他图灵机。")

if __name__ == "__main__":
    binary_increment()
    print()
    palindrome_checker()
```

### 核心概念图解

```
图灵机的基本结构：
┌──────────────────────────────────────────┐
│       无限纸带（内存）                    │
│  ...│ 1 │ 0 │ 1 │ 1 │ □ │ □ │...        │
│      ↑                                  │
│  ┌───┴───┐                              │
│  │ 读写头 │   ← 读/写/移动              │
│  └───┬───┘                              │
│      │                                   │
│  ┌───┴────────┐                          │
│  │ 有限状态控制器 │   ← "程序"           │
│  │ (规则表)     │                        │
│  └────────────┘                          │
└──────────────────────────────────────────┘

三个关键限制决定了可计算的边界：
① 纸带无限    →  内存无限（理论上）
② 状态有限    →  程序是有限的
③ 规则确定    →  每一步都由当前状态+符号唯一决定
```

**可计算性的层级：**

```
可计算问题                   不可计算问题
（存在图灵机能解决它）     （不存在图灵机能解决它）
        │                         │
   ┌────┴────┐               ┌────┴────┐
   │         │               │         │
  加法  回文检查        停机问题   ∀x P(x) 判定
  排序  字符串匹配      希尔伯特第十问题
  一切你能想到的算法     （丢番图方程可解性）
        │
        │  ── 丘奇-图灵论题 ──
        │  任何合理的计算模型都等价于图灵机
        ▼
  你的电脑 ≈ 图灵机 ≈ λ演算 ≈ 生命游戏 ≈ 规则 110
```

### 这个模式还出现在……

| 领域 | 描述 | 对应关系 |
|------|------|---------|
| **现代计算机 (von Neumann)** | CPU + 内存 + 指令集 | 图灵机的直接实现（纸带→内存，规则表→指令） |
| **λ 演算 (λ-calculus)** | 一个只有函数的计算模型 | 和图灵机等价——但完全不用"状态"，只用函数替换 |
| **生命游戏** | 四条规则的元胞自动机 | 图灵完全的——能在其中搭建出图灵机 |
| **编程语言** | Python/C/JS/Rust…… | 它们都是图灵完全的（只要内存不限） |

**拓展：停机问题——可计算性的边界**

核心证明——反证法（对角化）：

```
假设存在一台图灵机 H 能判定任何程序是否停机：
H(程序P, 输入I) → "停机" 或 "永不停机"

现在构造一个新程序 Q：
  Q(P) = 如果 H(P, P) 输出 "停机" → 进入死循环
         如果 H(P, P) 输出 "永不停机" → 停机

现在问：Q(Q) 是否停机？
  - 如果 H 说 Q(Q) 停机 → Q 进入死循环 → 不停机   矛盾
  - 如果 H 说 Q(Q) 不停机 → Q 停机 → 停机        矛盾

所以 H 不可能存在。
因此：不可能有通用算法判定任意程序是否停机。
```

这就是可计算性的**绝对边界**——不是"我们还不知道怎么做"，而是"在数学上不可能做到"。
